- 결과물이 바로바로 눈에 보이기 때문에 피드백하기 쉬웠다.
- 팀프로젝트 코드 작성에 속도가 중요하다는 걸 꺠달았다. 각자의 이해 속도도 다르고, 배경지식도 다르기 때문에, 계속 소통하면서 맞춰가는 것이 중요하다.

## 번외로 공부(기술부분)

```py
Get과 Post의 차이

"Get:" 서버에 정보를 조회하기 위해 사용되는 메소드
Ex) 게시판 글 조회
서버에 정보를 요청할 때, 쿼리스트링을 통해 데이터 전송

URL 주소에 ? 주소로 데이터를 전송, 길이 제한이 있음
보안에 중요한 데이터를 담아서 보내면 안됨(주소창에 뜨기 때문에)

GET의 경우 서버에 동일한 요청을 여러 번 보내더라도 항상 동일한 응답이 돌아와야 한다는 것을 의미.

"POST:" 서버에 데이터를 생성 또는 수정하기 위해 사용되는 메소드 ex) 게시판 글 작성 데이터는 body에 담아서 보냄.
데이터가 노출되지 않아 GET 방식보다는 안전함

POST의 경우 서버에 동일한 요청을 여러 번 보내더라도 응답 결과가 다를 수 있다는 것을 의미

"GET은 SELECT, POST는 CREATE, UPDATE에 가까움"
```

- HTTP와 HTTPS의 차이

```py
"HTTP:"
- 데이터를 요청/응답 시 사용되는 프로토콜
- 암호화되지 않은 평문 데이터를 전송하기 때문에 서버와 주고 받는 데이터를 볼 수 있음
- 80포트를 사용

"HTTP + SSL => HTTPS(Secure)"
- HTTPS에서 데이터 암호화가 추가된 프로토콜
- 암호화된 데이터를 전송하기 때문에 제 3자가 데이터를 볼 수 없음
- 443번 포트를 사용
```

- RESTful API

```py
- REST의 규칙을 지켜서 만든 API를 뜻함.
- 성능의 향상 보다는 규칙을 통한 API이해도와 호환성을 높이기 위함.
(유지보수 및 코드의 재사용성을 높일 수 있음)
```

- 동기와 비동기

```py
동기(Synchronous)방식:
- 요청과 결과가 동시에 일어남
- 요청을 하고 결과가 나올 떄까지 기다림(기다리기 때문에 작업의 순서를 보장)
- 여러 요청을 동시에 처리할 수 없음 ex) 1:1 고객 응대 카페
- Blocking 방식

비동기방식(Asynchronous):
- 요청과 결과가 동시에 일어나지 않는다.
- 요청 후 결과와 상관없이 다음 요청을 진행
- 여러 요청을 동시에 처리 ex) 1:N 고객 응대 카페
- Non-Blocking 방식
```

- 프록시(Proxy) 서버란?

```py
- proxy, 대리라는 뜻
- 클라이언트(유저)가 프록시 자신을 거쳐 다른 네트워크에 접속할 수 있도록 중간에서 대리역할을 하는 서버
- 즉, 인터넷에서 유저를 대신해서 데이터를 가져오는 서버

사용 이유:
- 캐시 데이터를 활용하기 위해, 프록시 서버에 요청된 내용은 굳이 한번 더 서버를 거칠 필요가 없음, 전송 시간을 아낄 수 있음
- 보안 목적, 프록시 서버로 중간 경유를 하게 되면 IP 노출을 하지 않아도 되기 때문, 이러한 이유로 프록시 서버를 방화벽 목적으로 사용하기도 함
```

- DNS

```py
- 도메인 이름과 IP 주소의 변환기 역할
- 192.0.0.1과 같은 숫자 IP주소를 사람이 읽을 수 있는 주소로 변환
```

- 쿠키와 세션의 차이

```py
쿠키: 브라우저 로컬에 저장되는 Key-Value 형태의 데이터 파일

세션: 브라우저가 종료되기 전까지 유저의 요청을 유지하게 해주는 기술

차이점:
1. 저장위치: 쿠키는 로컬에 저장, 세션은 로컬과 서버에 저장
2. 생명주기: 쿠키는 브라우저를 종료해도 파일로 남아있지만, 세션은 브라우저 종료시 삭제됨
3. 속도: 쿠키는 파일에서 읽기 때문에 상대적으로 빠름, 세션은 요청마다 서버에서 처리해야 하기 때문에 비교적 느림
```

- 프로세스와 스레드

```py
프로세스(Process): 운영체제로부터 자원을 할당받은 작업의 단위

스레드(Thread): 프로세스가 할당받은 자원을 이용하는 실행 흐름의 단위
```
